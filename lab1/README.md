DevOps Lab 1: Docker & Docker Compose Best Practices
Описание проекта
Учебный проект для демонстрации лучших и худших практик написания Dockerfile и Docker Compose. Приложение представляет собой простой Flask-сервер, отдающий приветственное сообщение.

1 часть:
Плохие практики в bad.Dockerfile
1. Использование тега latest
    В плохом Dockerfile используется FROM python:latest. Это плохая практика, потому что тег latest может измениться в любой момент. Сборка становится невоспроизводимой: сегодня всё работает, а завтра обновился базовый образ Python и приложение сломалось из-за несовместимости.
    В good.Dockerfile это исправлено на FROM python:3.11-slim. Фиксированная версия гарантирует одинаковую сборку в любое время.
2. Неправильный порядок COPY и RUN
    В плохом Dockerfile сначала выполняется COPY . ., а потом RUN pip install -r requirements.txt. Это плохая практика, потому что при любом изменении в коде (даже в одной строчке app.py) Docker сбрасывает кэш всех последующих слоёв и заново устанавливает все зависимости. Сборка становится медленной.
    В good.Dockerfile порядок исправлен: сначала копируется только requirements.txt, потом устанавливаются зависимости, и только потом копируется весь остальной код. Благодаря этому кэш зависимостей сохраняется, и при изменении кода команда pip install не запускается заново.
3. Запуск от имени root
    В плохом Dockerfile отсутствует создание пользователя, поэтому контейнер запускается от имени root по умолчанию. Это плохая практика, потому что контейнер работает с полными привилегиями. При уязвимости в приложении злоумышленник получит root-доступ внутри контейнера.
    В good.Dockerfile я добавил команды RUN useradd -m appuser и USER appuser. Это реализует принцип наименьших привилегий и ограничивает ущерб.

Две плохие практики эксплуатации контейнеров (Runtime)
1. Хранение данных внутри контейнера
    При удалении или пересоздании контейнера все данные, записанные внутри него, исчезнут безвозвратно. Это критично для баз данных и файлов пользователей.
    Чтобы этого избежать, надо использовать Volumes для монтирования данных на хост-машину.
2. Запуск с флагом --privileged
    Запуск с этим флагом отключает изоляцию безопасности контейнера. Контейнер получает доступ к устройствам хоста и может взломать весь сервер.
    Чтобы этого избежать, надо Выдавать контейнеру только нужные права. Не включать режим полной власти без причины.

2 часть:
Плохие практики в bad-compose.yml
1. Неявное указание Dockerfile
    В плохом Compose используется короткая запись build: .. Это не указывает явно, какой Dockerfile использовать. Меньше контроля над конфигурацией, сложнее управлять несколькими вариантами сборки.
    В good-compose.yml это исправлено на явное указание: build: context: . dockerfile: good.Dockerfile. Теперь понятно, какой файл используется для сборки.
2. Отсутствует политика перезапуска
    В плохом Compose нет директивы restart. При падении контейнера или перезагрузке сервера сервис не поднимется автоматически. Требуется ручное вмешательство.
    В good-compose.yml добавлено restart: unless-stopped. Контейнер автоматически перезапустится при сбое. Если же контейнер остановлен вручную, он не запустится сам после перезагрузки.
3. Нет изоляции сетей
    В плохом Compose нет раздела networks. Все сервисы автоматически попадают в одну общую сеть по умолчанию. Сервисы видят друг друга, что нарушает принцип минимальных прав и изоляции.
    В good-compose.yml созданы две отдельные сети (network_a и network_b). Каждый сервис подключен только к своей сети. Сервисы не могут пинговать друг друга. 
4. Нет переменных окружения
    В плохом Compose нет раздела environment. Конфигурация зашита в код приложения. Сложно менять поведение без пересборки образа. 
    В good-compose.yml добавлены переменные окружения, например SERVICE_NAME=web1. Это позволяет гибко настраивать поведение сервисов без изменения кода.

Сетевая изоляция
В good-compose.yml сервисы web1 и web2 не видят друг друга по сети, хотя запущены в одном проекте.
В разделе networks объявлены две отдельные сети: network_a и network_b. Сервису web1 назначена только сеть network_a. Сервису web2 назначена только сеть network_b
Принцип работы: Docker Compose создаёт две отдельные сети. web1 подключен только к network_a. web2 подключен только к network_b. Нет общего сетевого интерфейса между сервисами, значит нет маршрута между контейнерами. Команда ping не проходит.
