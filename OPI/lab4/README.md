🛠 РЕАЛИЗОВАННЫЙ ФУНКЦИОНАЛ (JMX MBEANS)



В проекте разработаны и зарегистрированы на платформенном сервере MBeans (пакет org.example.mbean):

💎 AreaStats — отслеживает общую статистику точек (попадания, промахи). Генерирует и отправляет JMX-уведомление "area.stats.three.misses" при фиксации 3 промахов подряд.

💎 ClickTiming — вычисляет средний интервал времени в миллисекундах между последовательными кликами пользователя по координатной плоскости.

💎 JmxRegistrar — компонент уровня приложения, отвечающий за автоматическую регистрацию MBean-классов в MBeanServer при старте.



🔍 ДИАГНОСТИКА И ПРОФИЛИРОВАНИЕ (ДО ОПТИМИЗАЦИИ)



В ходе анализа приложения с помощью Flame Graph, Call Tree (IntelliJ IDEA) и внешней утилиты VisualVM были локализованы две критические проблемы:

❌ Проблема №1 (Клиент): Поток main был непрерывно загружен. Около 40% процессорного времени уходило на Rhino (интерпретатор JavaScript в HttpUnit), который компилировал ненужные для теста скрипты. Также скорость ограничивал Thread.sleep(200).

❌ Проблема №2 (Сервер): Метод сервлета HelloWorld.doGet() тратил 74.4% времени на системные блокирующие вызовы ввода-вывода из-за пошаговой отправки HTML через 8 последовательных вызовов java.io.PrintWriter.println().



🚀 ПРОВЕДЕННАЯ ОПТИМИЗАЦИЯ (ПОСЛЕ)



В исходный код приложения были внесены следующие архитектурные исправления:

✅ Отключена поддержка скриптов на клиенте с помощью директивы setScriptingEnabled(false), что полностью убрало нагрузку от движка Rhino.

✅ Из бесконечного цикла клиента удалена искусственная задержка Thread.sleep(200) для оценки реальной пиковой пропускной способности.

✅ Множественные вызовы println() в сервлете заменены на эффективную буферизацию всей HTML-страницы в оперативной памяти через класс StringBuilder с однократной отправкой в сетевой поток.



📈 РЕЗУЛЬТАТЫ ОПТИМИЗАЦИИ



🔥 Нагрузка на обработку скриптов упала до уровня менее 1% (до 15 мс).

🔥 Тяжелые блокирующие вызовы PrintWriter.println полностью покинули топ ресурсоемких операций CPU.

🔥 Повторный мониторинг через вкладку Monitor в VisualVM и Call Tree в IDEA подтвердил успешное устранение вычислительных дефектов.
